## 二、整体技术架构分层模型

经过对所有项目的源码分析，可以抽象出一个**通用四层架构**：

```
┌─────────────────────────────────────────────────────────┐
│  L4 数据/智能层  (LLM、记忆、TTS/ASR、本地存储、云同步)     │
├─────────────────────────────────────────────────────────┤
│  L3 状态与逻辑层 (状态机、行为树、数值系统、IPC/WebSocket) │
├─────────────────────────────────────────────────────────┤
│  L2 输入采集层   (键鼠/手柄/系统资源/网络消息)             │
├─────────────────────────────────────────────────────────┤
│  L1 渲染与窗口层 (透明窗口 + 置顶 + 鼠标穿透 + 帧驱动)     │
└─────────────────────────────────────────────────────────┘
```

下面按照"从前往后"的顺序依次拆解。

## 三、L1：渲染与窗口层（动画驱动）

这是桌面宠物最特殊的部分——**它必须是一个无边框、透明背景、永远置顶、可点击穿透的窗口**。各项目实现差异很大。

### 3.1 窗口宿主：四种主流方案对比

#### 方案 A：Tauri 2（Rust + WebView，BongoCat 路线）

Tauri 用系统 WebView（Win 上是 WebView2，macOS 上是 WKWebView，Linux 上是 WebKitGTK），二进制小（10–20 MB），内存占用低。

`tauri.conf.json` 中关键窗口配置：

```json
{
  "windows": [{
    "transparent": true,
    "decorations": false,
    "alwaysOnTop": true,
    "skipTaskbar": true,
    "resizable": false,
    "visibleOnAllWorkspaces": true,
    "macOSPrivateApi": true
  }]
}
```

鼠标穿透通过 Rust 端调用：

```rust
window.set_ignore_cursor_events(true)?;
```

注意 `macOSPrivateApi: true` 是 macOS 上启用透明窗口的必需项。BongoCat 还专门用了 `tauri-nspanel` 插件来在 macOS 上获得自定义 NSPanel 行为（在全屏应用上方显示）。

#### 方案 B：Electron（Open-LLM-VTuber、chiikawa-pets 路线）

Electron 内置 Chromium，包体大（80–150 MB），但兼容性最好。BrowserWindow 配置：

```javascript
new BrowserWindow({
  transparent: true,
  frame: false,
  alwaysOnTop: true,
  skipTaskbar: true,
  hasShadow: false,
  resizable: false,
  webPreferences: { backgroundThrottling: false }
})
// 鼠标穿透
win.setIgnoreMouseEvents(true, { forward: true })
```

Open-LLM-VTuber 在 README 中明确把 "pet mode" 描述为"transparent, always-on-top Electron window"。

#### 方案 C：Qt（PySide6/PyQt，DyberPet 路线）

Qt 原生支持透明窗口，且不依赖 WebView，性能可观。DyberPet 的窗口实现：

```python
from PySide6.QtCore import Qt
from PySide6.QtWidgets import QWidget

self.setWindowFlags(
    Qt.FramelessWindowHint
    | Qt.WindowStaysOnTopHint
    | Qt.SubWindow            # 不在任务栏显示
    | Qt.Tool                  # macOS 不抢焦点
)
self.setAttribute(Qt.WA_TranslucentBackground)  # 透明背景
self.setAttribute(Qt.WA_TransparentForMouseEvents, False)
```

Qt 路线最大优势是内存占用低（一只宠物 30–60 MB），无 WebView 启动延迟，且 Qt 自带强大的动画框架 `QPropertyAnimation`。

#### 方案 D：Java Swing（Shimeji-ee 路线）

经典老牌方案。JFrame 配 `setUndecorated(true)` + `setBackground(new Color(0,0,0,0))`，借助 JNA 调用 Win32 `SetWindowLong` 实现穿透与置顶。优点是真跨平台、无需运行时分发；缺点是字体/HiDPI/macOS 适配有坑，已不被新项目采用。

### 3.2 角色渲染：三种主流技术

#### 路线 1：序列帧 PNG（Shimeji、DyberPet、WolfChen DesktopPet）

最朴素也最通用。每个动作就是一组编号 PNG（`stand1.png`、`stand2.png`...），通过定时器按帧切换。WolfChen DesktopPet 的 INI 配置是典型代表：

```ini
petaction=stand,walk,walkright,drag,falling
petactionnum=1,26,1,5,1     # 每个动作的帧数
petactionrate=0.95,0.05,0   # 触发概率
```

DyberPet 在此基础上更进一步，把 patpat 做成按饱食度等级分发的 dict：`{0:"actname", 3:"actname2"}`。

**渲染机制**：在 PySide6 中，DyberPet 用 `QLabel.setPixmap()` 配 `QTimer` 驱动；位移则用 `QPropertyAnimation(self.child, b"pos")` 这种基于 Qt 属性系统的插值动画。

#### 路线 2：Live2D（BongoCat、Open-LLM-VTuber、ai-live2d-go、bongo-cat-next）

Live2D 是日本 Live2D Inc. 的角色动画 SDK，分 Cubism 2.1 / 3 / 4 / 5 几代。它的优势是**用一张原画 + 网格变形 + 参数化动作**生成无限插值的 2D 动画，非常自然。

Web 端事实标准是 **`pixi-live2d-display`**（guansss 开发，BongoCat、bongo-cat-next 都在用），架构如下：

```
┌─────────────────────────────────────────────┐
│  Live2DModel  (PIXI.Container 的子类)        │
│   ├── InternalModel (抽象)                  │
│   │    ├── Cubism2InternalModel             │
│   │    └── Cubism4InternalModel             │
│   ├── MotionManager (动作管理)               │
│   ├── ExpressionManager (表情管理)           │
│   └── FocusController (视线跟随)             │
└─────────────────────────────────────────────┘
```

依赖三件套：
- **Cubism Core** (`live2dcubismcore.min.js`)：Live2D 官方运行时
- **PixiJS 6/7/8**：WebGL 渲染管线
- **pixi-live2d-display**：把 Live2D 模型包装成 PIXI 显示对象

驱动方式默认通过 `PIXI.Ticker.shared` 自动 `model.update(deltaTime)`；也可手动 `ticker.add(() => model.update(elapsedMS))` 接入自定义循环。

交互能力非常强：
- `model.focus(x, y)` —— 视线跟随鼠标
- `model.tap(x, y)` —— 点击触发对应区域动作
- `model.motion(group, index)` —— 播放指定动作组
- `model.expression(name)` —— 切换表情
- `model.on('hit', hitAreas => ...)` —— 命中区域回调

Open-LLM-VTuber 的玩法更高级：后端用 LLM 输出包含情感关键词的文本，前端 frontend 把关键词映射到 expression（在 `model_dict.json` 中配置 `tapMotions: { body: { tap_body: 30, shake: 30 }, head: { flick_head: 40 } }` 这种带权重的多动作）。

#### 路线 3：游戏引擎渲染（Agentic-Desktop-Pet 用 Godot；CodeWalkers 用 DOM）

Agentic-Desktop-Pet 直接用 Godot 4 来做前端，把宠物当成游戏 Sprite 处理，自带骨骼动画、物理、状态机。代价是包体大、与桌面集成需要额外工作（透明窗口要 Godot 4.x 才支持得好）。

CodeWalkers 是另一个极端——直接用 DOM。它的关键性能优化（dev.to 文章中明确提到）：
> 不把坐标 (x,y) 放进 React State 触发 re-render，而是用 ref 直接操作 DOM 的 `transform` 属性，配 `requestAnimationFrame`，硬撑出 60 FPS 原生流畅度。

这是 React/Web 桌宠的标准性能套路，值得在自研时复用。

### 3.3 帧驱动循环对比

| 技术栈 | 主循环 | 触发频率 | 备注 |
|---|---|---|---|
| PixiJS | `PIXI.Ticker` | 60 FPS（vsync） | 自动 |
| 原生 Web | `requestAnimationFrame` | 与刷新率同步 | 最省电 |
| PySide6 | `QTimer` 或 `QPropertyAnimation` | 自定义 | 通常 30–60 FPS |
| Java Swing | `javax.swing.Timer` | 自定义 | Shimeji 用 ~25 FPS |
| Godot | `_process(delta)` | 引擎主循环 | 与 vsync 绑定 |

## 四、L2：输入采集层（与桌面/系统的桥梁）

桌面宠物要"活"，必须能感知用户。所有项目都做了**全局键鼠 hook**——这恰恰是 sandbox 化框架的难点。

### 4.1 全局键鼠 Hook

#### Rust 生态：`rdev` 与替代品

BongoCat、KeyKey、大部分 Tauri 桌宠都用 `Narsil/rdev`：

```rust
use rdev::{listen, Event, EventType, Key};

fn callback(event: Event) {
    match event.event_type {
        EventType::KeyPress(key) => { /* emit to webview */ }
        EventType::ButtonPress(button) => { ... }
        EventType::MouseMove { x, y } => { ... }
        _ => {}
    }
}

// 在独立线程中阻塞监听
std::thread::spawn(|| {
    if let Err(error) = listen(callback) {
        eprintln!("Error: {:?}", error);
    }
});
```

**`rdev` 的关键限制**（项目中有踩坑记录）：
- macOS 必须授予 Accessibility 权限，否则 callback 静默不触发
- 进程必须是父进程（fork 后会失效）
- Tauri 2 上有个已知 bug：当主窗口获得焦点时，rdev 在同一进程内会停止收到键盘事件（鼠标仍正常）—— GitHub issue #14770。一些项目因此把 hook 拆到独立子进程。

替代方案：
- `gilrs`：BongoCat 用它处理手柄输入（按键 + 摇杆）
- `mouce` / `mouse_position`：CrabNebula 教程项目使用
- 自研 Win32 hook：chiikawa-pets 直接写了 C++ 模块，性能比 rdev 好（按作者说法），通过 N-API 接给 Electron

#### Python 生态：`pynput`

DyberPet 用 `pynput==1.7.6`：

```python
from pynput import keyboard, mouse

def on_press(key):
    # 触发桌宠交互
    pass

listener = keyboard.Listener(on_press=on_press)
listener.start()  # 非阻塞
```

`pynput` 跨平台开箱即用，但是同样需要 macOS 的辅助功能权限。

#### Electron 生态

Electron 自身没有全局 hook，常见做法：
- 用 Node 原生模块 `iohook`（已不维护）/`uiohook-napi`
- 自研 C++ + N-API（如 chiikawa-pets）
- 用 `globalShortcut` —— 但只能注册有限组合键，不能监听所有按键

### 4.2 鼠标位置 / 屏幕信息

- Rust: `mouse_position` crate / `display-info`
- Python: `pyautogui.position()` / Qt 的 `QCursor.pos()`
- Web 内（Live2D 视线跟随）：`window.addEventListener('mousemove', e => model.focus(e.clientX, e.clientY))`

### 4.3 系统资源采集（CPU、内存、网络）

部分项目把宠物状态绑到系统状态上（比如 CPU 高了就显示"累"）。这部分技术栈很简单：
- Python: `psutil`
- Rust: `sysinfo` crate
- Node: `systeminformation`

## 五、L3：状态与逻辑层（行为驱动）

这是桌宠从"会动的图"变成"有性格"的核心。

### 5.1 行为驱动模型对比

#### 模型 A：XML 行为树（Shimeji-ee）

经典老派方案。`actions.xml` 定义"原子动作"（怎么播放序列帧），`behaviors.xml` 定义"在什么状态下选什么动作"。例如：

```xml
<!-- actions.xml -->
<Action Name="Walk" Type="Move">
  <Animation>
    <Pose Image="/shime2.png" Velocity="-2,0" Duration="4"/>
    <Pose Image="/shime3.png" Velocity="-2,0" Duration="4"/>
  </Animation>
</Action>

<!-- behaviors.xml -->
<Behavior Name="Walk" Frequency="100" Hidden="false"/>
<Behavior Name="Sit" Frequency="50"/>
```

**必备 Action**：`ChaseMouse`、`Fall`、`Dragged`、`Thrown`。这套约定俗成的 schema 至今仍是桌宠"行为契约"的事实标准。优点：纯数据驱动，无代码即可创建新角色；缺点：表达能力受限于 XML 语法，复杂逻辑（条件、随机、组合）写起来繁琐。

#### 模型 B：JSON 配置 + 状态机（DyberPet、WolfChen DesktopPet）

DyberPet 用 `pet_conf.json` 管角色，`item_config.json` 管物品，`note_config.json` 管通知。它的数值系统包括：
- 饱食度（每分钟下降，触发不同动作组）
- 好感度
- 物品/背包系统
- 待办事项 + 番茄钟

行为选择是基于权重和数值等级的随机+条件混合，比纯 Shimeji 灵活。

#### 模型 C：基于 Composable / Hooks（BongoCat）

BongoCat 没有显式的"行为引擎"，而是用 Vue 3 Composition API 把"输入 → 动画"做成响应式 pipeline：

```
useDevice()  ── 监听 Tauri 后端 emit 的 DEVICE_CHANGED 事件
useGamepad() ── 处理手柄事件
     ↓ 规范化后的输入对象
useModel()   ── 把输入映射成 Live2D 参数更新
     ↓
PixiJS 渲染（60 FPS）
```

状态全部存在 Pinia store 里，并通过 `@tauri-store/pinia` 自动持久化到磁盘。包括：
- `useAppStore`（应用元数据）
- `useModelStore`（模型配置）
- `useCatStore`（猫和窗口设置）
- `useGeneralStore`（外观与通用设置）
- `useShortcutStore`（全局快捷键）

#### 模型 D：LLM 驱动 Agent（Open-LLM-VTuber、Agentic-Desktop-Pet、ai-live2d-go）

最前沿的路线。让 LLM 直接输出**结构化的情感/动作标签**，前端解析后驱动模型。Open-LLM-VTuber 的做法：

1. 在 system prompt 中注入可用情感关键词（从 `model_dict.json` 读取，例如 `[joy] [angry] [shy]`）
2. LLM 流式输出文本，包含这些标签
3. 后端 expression extraction system 提取标签，通过 WebSocket 推给前端
4. 前端切换 Live2D 表情 + 播放动作

ai-live2d-go 更进一步，做了 ReAct Loop（最多 10 轮工具调用）和长期记忆：

```
User Message → Mode Router → System Prompt (persona + memory + manual)
                              ↓
                            LLM
                          ↓     ↓
             Text reply        tool_calls → execute → feed back → re-request
                                                      ↑___________|  (ReAct)
                                                     ↓
        Leave conversation → Summarize → Memory refinement → SQLite
```

Agentic-Desktop-Pet 的目录布局直接表达了"下一代桌宠"的组件构成：

```
backend/learn_agent/
├── agent/        # 主 Agent + 记忆管理
├── memory/       # Cognee 记忆系统
├── emotion/      # 情感引擎
├── rpg/          # RPG 属性
├── llm/          # LLM 接口
└── tool/         # file_tool / code_tool / todo_tool
```

### 5.2 IPC：前后端怎么通信

| 项目 | 协议 | 何时用 |
|---|---|---|
| BongoCat | Tauri `invoke` (命令) + `emit/listen` (事件) | 输入事件高频用 emit，配置读写用 invoke |
| chiikawa-pets | Electron `ipcMain.handle` + `ipcRenderer.invoke` | 同上，事件用 `webContents.send` |
| Open-LLM-VTuber | **WebSocket** (`/client-ws`) | 前后端解耦，桌面端和 Web 端共用一个后端 |
| Agentic-Desktop-Pet | HTTP REST + WebSocket | Godot ↔ Python FastAPI |
| DyberPet | 进程内 Qt 信号槽 | 单进程，无跨语言 IPC |

**WebSocket 路线的特别之处**：Open-LLM-VTuber 的 `WebSocketServer` 是一个 FastAPI app，每个客户端连接拿到一个 `ServiceContext.clone()` 实例（持有独立的 ASR/TTS/LLM/Live2D 引擎引用）。这样既能单后端服务多前端，也能让"Web 模式"和"桌面 pet 模式"共用同一套逻辑。

## 六、L4：数据与智能层（后台接入）

这一层直接决定了桌宠是"玩具"还是"助手"。

### 6.1 本地持久化

| 方式 | 代表项目 | 适用场景 |
|---|---|---|
| JSON 平铺文件 | DyberPet (`act_data.json`、`pet_data.json`)、Shimeji (`settings.properties`) | 配置 + 小规模数值 |
| SQLite | ai-live2d-go (`memory_fragments` 表)、Agentic-Desktop-Pet | 对话历史、记忆碎片 |
| Pinia + 自动序列化 | BongoCat (`@tauri-store/pinia`) | 前端响应式状态自动落盘 |
| Electron Store | chiikawa-pets 类项目 | 用 nedb / json 文件 |

DyberPet 还专门处理了"数据存写大概花费 0.43ms，有极小概率在该间隔内关闭软件导致数据丢失"——在退出前主动 flush + 冻结数据，避免脏写。这是工程细节中容易忽视的一点。

### 6.2 LLM 接入

主流接入模式三种：

#### 1. OpenAI Compatible API（最通用）

ai-live2d-go 在 `electron/ai.config.ts` 中支持 OpenAI / DeepSeek / Zhipu / Moonshot / Doubao / Qwen / SiliconFlow，全都用 OpenAI 兼容协议。这是目前最佳实践——一份代码接所有云端 LLM。

#### 2. 本地推理

Open-LLM-VTuber 直接列出支持：Ollama、GGUF、LM Studio、vLLM。本地推理路线的好处是隐私、离线、无成本，但需要用户有 GPU。

#### 3. CLI 包装

CodeWalkers 走的是另一条路：直接调用 GitHub Copilot CLI 和 Gemini CLI，把 stdin/stdout 通过 PTY (`tauri-plugin-shell` + 伪终端) 接到一个气泡 UI 里。这个思路对开发者类桌宠尤其合适。

### 6.3 ASR / TTS（Open-LLM-VTuber 路线）

要让桌宠能听能说，技术栈非常重：

```
ASR:
  本地: sherpa-onnx, FunASR, Faster-Whisper, Whisper.cpp
  云端: Groq Whisper, Azure ASR

TTS:
  本地: sherpa-onnx, pyttsx3, MeloTTS, Coqui-TTS, GPTSoVITS, Bark, CosyVoice
  云端: Edge TTS, Fish Audio, Azure TTS
```

值得注意的是，所有项目都通过**接口抽象**（`ASRInterface`、`TTSInterface`）+ **工厂模式**做切换，YAML 配置文件指定具体实现。这是模块化设计的成熟范式。

### 6.4 长期记忆 / RAG

- **Letta**（前 MemGPT）：Open-LLM-VTuber 用它做长期记忆，让桌宠记住用户偏好和历史对话
- **Cognee**：Agentic-Desktop-Pet 用它做语义记忆图
- **简单总结**：ai-live2d-go 每 10 轮对话生成 150 字摘要写入 SQLite，再用 "Hermes-style" 结构化提炼成 USER profile / MEMORY config

### 6.5 网络数据 / 工具调用

成熟路线已经形成：通过 LLM 的 tool calling，桌宠可以：
- 读写文件（`file_tool`）
- 执行代码（`code_tool`，沙箱化）
- 管理 Todo（`todo_tool`）
- 桥接到 IM（ai-live2d-go 的 `bridges/discord`、`bridges/wechat`）

Open-LLM-VTuber 还集成了 BiliBili 直播 SDK，能接弹幕互动；以及"浏览器控制"能力，桌宠可以自己开浏览器执行任务。

## 八、推荐技术选型（按目标场景）

### 场景 1：轻量交互型桌宠（响应键盘鼠标）
**推荐**：Tauri 2 + Vue/React + PixiJS + pixi-live2d-display + rdev
**参考**：BongoCat
**理由**：包体最小、性能足够、Live2D 生态成熟、跨平台

### 场景 2：AI 助手型桌宠（带对话/语音）
**推荐**：Electron 或 Tauri + 后端 FastAPI + WebSocket + Live2D
**参考**：Open-LLM-VTuber、ai-live2d-go
**理由**：Python 后端处理 LLM/ASR/TTS 最方便，前后端解耦能让一份后端服务 Web 和桌面两个客户端

### 场景 3：框架化（让别人创建模组）
**推荐**：PySide6 + JSON 配置 + 序列帧
**参考**：DyberPet
**理由**：素材门槛低（只要画 PNG），Python 生态丰富，Qt 内存占用小

### 场景 4：复杂游戏化桌宠（养成、RPG）
**推荐**：Godot 4 + Python Agent 后端
**参考**：Agentic-Desktop-Pet
**理由**：游戏引擎自带物理、状态机、动画系统

### 场景 5：极简实验/学习项目
**推荐**：PyQt + GIF
**参考**：Pet-GPT、WolfChen DesktopPet
**理由**：百行代码就能跑起来

## 九、工程踩坑清单（来自源码与 issues）

1. **macOS 权限**：`rdev`、`pynput` 都需要 Accessibility 权限，必须在 README 中明确指引用户去系统设置开启
2. **Tauri 主窗口聚焦时 rdev 键盘事件丢失**（issue #14770）：解决办法是把 hook 放独立子进程
3. **透明窗口 + 鼠标穿透**：所有平台都不是开箱即用——Linux X11 需要 compositor 支持，Wayland 兼容性更差；macOS 需要 `macOSPrivateApi: true`
4. **Live2D 模型协议**：Cubism 2.1 用 `.model.json`，Cubism 3+ 用 `.model3.json`，不向后兼容 —— Open-LLM-VTuber v1.2.0 干脆移除了 2.1 支持
5. **WebView 透明区域仍然吃点击**：Tauri issue #13070 提到这个限制，要么用多 WebView 分块，要么 set_ignore_cursor_events 整窗口穿透（但这样宠物本身也点不到了，需要在 JS 里动态切换）
6. **数据写入丢失**：DyberPet 在退出前主动 flush + 冻结的做法值得学习
7. **DOM 动画性能**：永远用 ref + transform + RAF，不要把坐标进 React state（CodeWalkers 经验）
8. **Pixi.js + Webpack 双包问题**：pixi-live2d-display 文档里专门强调过，要小心 Pixi 模块被打包两次
